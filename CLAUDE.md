# CodePulse

GitHub App that monitors codebase health on every push.
A BullMQ pipeline fans out to 5 parallel analysis workers, aggregates results into a
weighted HealthScore (0–100), and streams live updates to a Next.js dashboard.
Unique feature: architectural drift detection via pgvector cosine similarity.

## ORM — Prisma

All database access goes through Prisma. No raw SQL except vector operations.

**Prisma client singleton (`api/lib/prisma.js`):**
```javascript
const { PrismaClient } = require('@prisma/client');

const globalForPrisma = global;
const prisma = globalForPrisma.prisma ?? new PrismaClient();
if (process.env.NODE_ENV !== 'production') globalForPrisma.prisma = prisma;

module.exports = prisma;
```

**Neon requires two URLs in `prisma/schema.prisma`:**
```prisma
datasource db {
  provider  = "postgresql"
  url       = env("DATABASE_URL")    // pooled (pgBouncer) — all runtime queries
  directUrl = env("DIRECT_URL")      // direct — prisma migrate only
}

generator client {
  provider = "prisma-client-js"
}
```

**Why two URLs:** Neon routes queries through pgBouncer (connection pooler) by default.
Prisma migrations cannot run over pooled connections — `DIRECT_URL` bypasses pgBouncer
for `prisma migrate deploy` only. The app always uses `DATABASE_URL` (pooled).

**Common Prisma commands:**
```bash
npx prisma migrate dev        # create + apply migration locally
npx prisma migrate deploy     # apply migrations in production (Render)
npx prisma generate           # regenerate client after schema change
npx prisma studio             # DB browser UI at localhost:5555
npx prisma db push            # push schema without migration (prototyping only)
```

## Prisma schema

```prisma
datasource db {
  provider  = "postgresql"
  url       = env("DATABASE_URL")
  directUrl = env("DIRECT_URL")
}

generator client {
  provider = "prisma-client-js"
}

model Repo {
  id             String         @id @default(cuid())
  githubRepoId   Int            @unique
  owner          String
  name           String
  installationId Int
  userId         String
  createdAt      DateTime       @default(now())
  snapshots      Snapshot[]
  fileAnalyses   FileAnalysis[]
  digestLogs     DigestLog[]
}

model Snapshot {
  id           String         @id @default(cuid())
  repoId       String
  repo         Repo           @relation(fields: [repoId], references: [id], onDelete: Cascade)
  commitSha    String
  healthScore  Float
  complexity   Float
  vulnCount    Int
  deadCode     Float
  coverage     Float
  driftScore   Float
  createdAt    DateTime       @default(now())
  fileAnalyses FileAnalysis[]

  @@index([repoId, createdAt])
}

model FileAnalysis {
  id          String    @id @default(cuid())
  repoId      String
  repo        Repo      @relation(fields: [repoId], references: [id], onDelete: Cascade)
  snapshotId  String?
  snapshot    Snapshot? @relation(fields: [snapshotId], references: [id])
  filePath    String
  complexity  Float     @default(0)
  isDead      Boolean   @default(false)
  driftScore  Float?
  embedding   Unsupported("vector(1536)")?
  createdAt   DateTime  @default(now())

  @@index([repoId])
  @@index([repoId, filePath])
}

model DigestLog {
  id      String   @id @default(cuid())
  repoId  String
  repo    Repo     @relation(fields: [repoId], references: [id], onDelete: Cascade)
  sentAt  DateTime @default(now())
  summary Json
}
```

**pgvector — run once in Neon SQL editor after first deploy:**
```sql
CREATE EXTENSION IF NOT EXISTS vector;
CREATE INDEX ON file_analysis USING hnsw (embedding vector_cosine_ops);
```

Note: `Unsupported("vector(1536)")` is the correct Prisma syntax for pgvector columns.
Prisma doesn't natively know the vector type — we use `Unsupported()` and handle
all vector reads/writes with raw SQL via `prisma.$executeRaw` and `prisma.$queryRaw`.

**Inserting an embedding:**
```javascript
await prisma.$executeRaw`
  UPDATE file_analysis
  SET embedding = ${JSON.stringify(vector)}::vector
  WHERE id = ${id}
`;
```

**Cosine similarity query:**
```javascript
const results = await prisma.$queryRaw`
  SELECT file_path, 1 - (embedding <=> ${JSON.stringify(queryVector)}::vector) AS similarity
  FROM file_analysis
  WHERE repo_id = ${repoId}
  ORDER BY similarity DESC
  LIMIT 5
`;
```

## Critical rules — always follow these

### Prisma
- All DB queries use Prisma except vector operations (use `$executeRaw` / `$queryRaw`)
- Always run `npx prisma generate` after any schema change before starting the app
- `DATABASE_URL` (pooled) for all runtime queries — never use `DIRECT_URL` in app code
- `Unsupported("vector(1536)")` on the embedding column is correct — do not change it
- Add `?connect_timeout=30&pool_timeout=30` to both Neon URLs to handle cold starts

### pgvector
- Embedding model: `text-embedding-3-small` — 1536 dimensions — do not change
- Cosine distance operator: `<=>` — `1 - (embedding <=> $vector)` = cosine similarity
- Drift threshold: similarity **< 0.72** = architectural drift flagged
- Chunking strategy: by function/class boundary — not arbitrary line count
- Add HNSW index AFTER initial bulk embedding completes — not before

### Docker / Local Postgres
- Local dev uses `pgvector/pgvector:pg16` image — never `postgres:latest`
- Plain Postgres does not include the vector extension

### BullMQ
- Worker is always a **separate process** from API — never import worker logic into `api/index.js`
- Always call `worker.close()` on SIGTERM — lets current job finish gracefully
- Fan-out via `addBulk` in `api/services/queue.js` — never loop individual `queue.add()` calls
- Queue names: `complexity-queue` `deadcode-queue` `vuln-queue` `coverage-queue` `drift-queue`
- Concurrency: `5` per worker
- Retry: `attempts: 5, backoff: { type: 'exponential', delay: 2000 }`

### GitHub App / Octokit
- Never cache the Octokit instance — installation tokens expire every hour
- Call `app.getInstallationOctokit(installationId)` fresh on every request
- File content from GitHub API is base64 — decode: `Buffer.from(data.content, 'base64').toString('utf-8')`
- `api/middleware/verifyGithub.js` — never modify this file
- Use `crypto.timingSafeEqual` for signature comparison — never `===`

### Render free tier
- Web Service spins down after 15 min of inactivity — Background Worker does not
- Add `GET /health` route returning `{ ok: true }`
- Set up UptimeRobot (free) to ping `/health` every 10 min to prevent spindown

### Code style
- Backend: CommonJS (`require` / `module.exports`) — not ESM
- Dashboard: TypeScript strict mode
- One file per worker type in `worker/workers/`
- Wrap all worker logic in try/catch — throw to mark job failed, never crash the process

## HealthScore formula

```
HealthScore (0–100) = 100
  - (complexity_penalty  × 0.25)   // escomplex score normalized 0–100
  - (vuln_penalty        × 0.25)   // critical=10pts, high=5pts, moderate=2pts
  - (deadcode_penalty    × 0.15)   // (unused exports / total exports) × 100
  - (coverage_penalty    × 0.20)   // (1 - coverage_ratio) × 100
  - (drift_penalty       × 0.15)   // (flagged files / total files) × 100
```

## Bull Board

Queue inspector UI mounted at `/admin/queues` on the API server.
Access locally at `http://localhost:3000/admin/queues`.

```javascript
const { createBullBoard } = require('@bull-board/api');
const { BullMQAdapter } = require('@bull-board/api/bullMQAdapter');
const { ExpressAdapter } = require('@bull-board/express');

const serverAdapter = new ExpressAdapter();
serverAdapter.setBasePath('/admin/queues');
createBullBoard({
  queues: [
    new BullMQAdapter(complexityQueue),
    new BullMQAdapter(deadcodeQueue),
    new BullMQAdapter(vulnQueue),
    new BullMQAdapter(coverageQueue),
    new BullMQAdapter(driftQueue),
  ],
  serverAdapter,
});
app.use('/admin/queues', serverAdapter.getRouter());
```

## Drift detection — how it works

1. **At install time** — one-time bulk embedding job:
   - Fetch every file via GitHub API
   - Chunk by function/class boundary
   - Embed with `text-embedding-3-small` (1536 dims)
   - Store in `file_analysis.embedding` via `prisma.$executeRaw`

2. **On each push** — drift worker:
   - Embed each new/modified file
   - Run pgvector cosine similarity query (top 5 most similar existing files)
   - `MAX(similarity)` = drift score for this file
   - Score < 0.72 → flag as architectural drift

3. **HNSW index** — add after initial bulk embed:
   ```sql
   CREATE INDEX ON file_analysis USING hnsw (embedding vector_cosine_ops);
   ```

## Deployment checklist

### 1. Neon (do this first)
- [ ] Create account at neon.tech → new project → Postgres 16
- [ ] Copy pooled connection string → `DATABASE_URL`
- [ ] Copy direct connection string → `DIRECT_URL`
- [ ] Append `?connect_timeout=30&pool_timeout=30` to both strings
- [ ] Neon SQL editor → `CREATE EXTENSION IF NOT EXISTS vector;`

### 2. Render
- [ ] Create Web Service → repo → build: `npm install` → start: `node api/index.js`
- [ ] Create Background Worker → same repo → start: `node worker/index.js`
- [ ] Create Redis instance → copy `REDIS_URL`
- [ ] Set all env vars on both services (copy from .env.example)
- [ ] Copy Web Service URL (e.g. `https://codepulse-api.onrender.com`)
- [ ] Set up UptimeRobot → ping `https://codepulse-api.onrender.com/health` every 10 min

### 3. GitHub App
- [ ] Webhook URL: `https://codepulse-api.onrender.com/webhook`
- [ ] Events: Push
- [ ] Permissions: Contents (read), Metadata (read)
- [ ] Install App on a test repo

### 4. Vercel
- [ ] Import `dashboard/` directory
- [ ] Set `NEXT_PUBLIC_API_URL` = Render Web Service URL
- [ ] Set Clerk env vars
- [ ] Deploy → copy Vercel URL → set as `FRONTEND_URL` in Render env vars

### 5. Post-deploy
- [ ] `npx prisma migrate deploy` (run from local with production `DATABASE_URL` + `DIRECT_URL`)
- [ ] Confirm pgvector extension enabled in Neon SQL editor
- [ ] Push to test repo → webhook fires → check Bull Board at `/admin/queues`
- [ ] After first bulk embedding job: add HNSW index in Neon SQL editor
